# Healthcare RAG Chat System — Plan

> A guarded, tiered Retrieval-Augmented-Generation chatbot for the Medelite
> healthcare context. Answers **structured** questions from the SQL facility
> warehouse and **unstructured** medical questions from a PubMed knowledge base,
> with security + out-of-domain gating, a rule→embedding→web-search router, and
> grounded answers generated by Claude.
>
> Source brief: `anushka.txt`. Audience: us, building it sequence-wise.

---

## 0. Locked decisions (from the brainstorm)

| Concern | Decision |
|---|---|
| Corpus | PubMed / medical literature (abstracts, guidelines) |
| Embeddings | `pritamdeka/S-PubMedBert-MS-MARCO` (PubMedBERT family, sentence-tuned for retrieval) |
| Vector store | **SQLite + `sqlite-vec`** — unified with the existing warehouse DB |
| Structured questions | Routed to the **SQL facility warehouse** we already built |
| Unstructured questions | RAG over the PubMed vector store |
| Security gate | rule-based (regex) → embedding-based (similarity to unsafe/off-topic centroid) |
| Domain gate | rule-based keywords → embedding similarity |
| Intent router | T1 rule-based (direct/structured→SQL) → T2 embedding RAG → T3 web-search fallback |
| RAG→web fallback | top cosine similarity `< threshold` → web search |
| Response | **2-stage**: (a) extractive context compression (shorten), (b) Claude generates grounded answer + citations |
| Generation model | **Claude Opus 4.8** (`claude-opus-4-8`) via the official `anthropic` SDK, streaming |
| Web search | Claude server-side `web_search` tool (no separate search API) |
| Backend | **Python FastAPI** service hosting the pipeline + calling Claude server-side |
| Frontend | **React** Chat tab in the existing app, talking to the FastAPI backend |
| Corpus source | **PubMedQA sample** (`chat/data/pubmedqa_sample.json`) — fully offline demo fixture |
| Embedding stack | **Biomedical local**: `S-PubMedBert-MS-MARCO` via `sentence-transformers` (load once at startup) |
| Build scope | **Full pipeline, all phases** (both gates → tiered router → web search → 2-stage response → FastAPI → React) |

**Why a Python backend (refinement of "Netlify proxy"):** the PubMedBERT embedding
model and vector search live in the Python ML ecosystem. So instead of a Node
Netlify function, a small FastAPI service hosts the whole pipeline *and* makes the
Claude calls server-side (API key never reaches the browser). The React app calls
this backend. This also keeps the chat code next to the ETL/warehouse code.

## 0a. Brainstorm resolutions (refinements to lock above)

These tighten the locked decisions after a second brainstorm pass. They change *how*
each stage behaves, not the overall architecture.

- **PubMedQA is both corpus and free eval set.** Each entry carries a question,
  context(s), a PMID, and a `final_decision`. Split it: the **majority** of entries →
  the RAG corpus (their contexts chunked + embedded); a **held-out slice** → an eval
  set that proves retrieval (does an entry's own context come back for its question?).
  No manual labeling needed.
- **Calibrate `RAG_THRESHOLD`, don't guess it.** The current `0.55` is a placeholder.
  Using the held-out questions, compare top cosine to the *true* context vs. to random
  contexts and pick the value that separates them. Ship the measured number + the
  method. (Keep it env-overridable.)
- **Out-of-domain policy (resolves the gate contradiction):**
  - security gate fail ⇒ **refuse** (safe message);
  - out-of-domain (not healthcare at all) ⇒ **refuse politely**;
  - in-domain **but** top RAG similarity `< threshold` ⇒ **web search**.
  Web search is the "on-topic but corpus doesn't cover it" escape hatch, *not* a
  catch-all for off-topic queries.
- **T1 structured = parameterized SQL templates, not text-to-SQL.** Rule matches
  (6-digit CCN; words like `rating / rank / beds / completeness / history`) select a
  fixed, parameterized query over the warehouse. Deterministic, no injection surface,
  easy to unit-test. (Text-to-SQL explicitly out of scope.)
- **Real citations via PMIDs.** Thread each chunk's PMID through `store.py` metadata so
  Claude cites concrete `PMID:…` sources, not just chunk numbers.
- **Model loads once.** Instantiate `S-PubMedBert` a single time (FastAPI lifespan /
  ingest singleton), never per request — PubMedBERT cold start is non-trivial.
- **Stream the answer, trace the route.** Only stage 4b (Claude) streams. The SSE
  contract also emits a small **trace event** — which tier fired, top similarity,
  citations — so the React UI can "show its work" (the demo money-shot).

## 0b. Phase 1 resolutions (nailed down against the scaffold)

Verified against the on-disk code + data (40 PubMedQA records → ~128 chunks). The
store/metadata layer is **already built and passing tests**; these lock the rest.

**Already done (don't rebuild):**
- `store.py` — `chunks` (pmid, question, section, text, long_answer, source_url) +
  `vec_chunks` (sqlite-vec `float[768] distance_metric=cosine`). Cosine wired
  correctly: vectors L2-normalized in `embeddings.py`, similarity = `1 − distance`.
  `test_store.py` (fake-vector, offline) passes.
- **PMID citations already threaded** — `source_url` = `…/pubmed/{pmid}/`. (Supersedes
  the earlier "remaining: PMID metadata" note in the checklist.)
- **Load-once model** — `embeddings.get_model()` lazy singleton.

**Chunking — keep per-paragraph; drop the "512-token window + overlap" idea.**
PubMedQA contexts are already short, section-labeled paragraphs
(BACKGROUND/METHODS/RESULTS). One chunk per `CONTEXTS` paragraph, carrying its
`LABELS` section, is the natural unit; the generic 512-token-window text does not fit
this corpus.

**Eval split is *analytical*, not a physical hold-out (key decision).** With only 40
records, removing any from the index shrinks a tiny corpus and complicates `ingest.py`.
So **ingest all 40**, and let `eval.py` simulate coverage by filtering on PMID
in-memory. Two metrics:
- **Retrieval quality** (validates store+search): for each record's `QUESTION`, run
  `store.search(k)`; check whether a chunk from its *own* PMID comes back ⇒
  **recall@k** + **MRR**. (Own context is in-corpus, so near-perfect; proves the
  pipeline, MRR catches ranking regressions.)
- **Threshold calibration** (sets `RAG_THRESHOLD`): per question compute
  *covered* = max sim over **all** chunks (own context present, should be **high**) and
  *uncovered* = max sim over chunks **excluding its own PMID** (simulates an uncovered
  query, should be **lower**). The separation of the two distributions yields a
  *measured* threshold, replacing the placeholder `0.55`. No DB rebuild — just
  exclude-own-PMID in the scan.
- **Threshold rule:** pick the value that **maximizes (covered-accept − uncovered-accept)**
  — i.e. directly optimizes the route decision (vs. midpoint-of-means or a percentile).

**Small hardening (in Phase 1):**
- Assert `vectors.shape[1] == EMBED_DIM` at ingest so a model swap fails loudly, not
  silently against the `float[768]` table.
- **Store `final_decision` + `YEAR`** on chunks (currently unused) so the answer stage
  can cite a study's actual conclusion ("this 2011 study found *yes*…").

**Remaining Phase 1 work:** `eval.py` (two metrics + recommended threshold CLI),
`test_eval.py` (offline, fake-vector metric math), the dim assertion + the two metadata
columns, and **one real ingest to produce the threshold number** — note this downloads
~400 MB (PubMedBERT isn't cached yet), the one slow step.

## 1. Architecture

```
                         React Chat UI  (streaming)
                                │  POST /chat  (SSE stream back)
                                ▼
        ┌──────────────── FastAPI backend (Python) ─────────────────┐
        │                                                           │
        │  1) SECURITY GATE   rule-based → embedding-based          │
        │  2) DOMAIN GATE     rule-based → embedding-based          │
        │  3) INTENT ROUTER                                         │
        │       T1 rule  → STRUCTURED → SQL over warehouse.db       │
        │       T2 embed → RAG → retrieve from vector store         │
        │       T3 web   → Claude web_search (fallback)             │
        │            (T2 sim < threshold ⇒ drop to T3)              │
        │  4) RESPONSE                                              │
        │       a) SHORTEN  extractive compression of context       │
        │       b) ANSWER   Claude Opus 4.8 + citations (stream)    │
        └───────────────────────────┬───────────────────────────────┘
                                     │
        ┌──────────── SQLite (one file) ────────────┐
        │  warehouse tables (Phase B/C/D work)       │
        │  + vec_chunks   (sqlite-vec embeddings)    │
        │  + chunks       (text + source metadata)   │
        └────────────────────────────────────────────┘
```

The chat pipeline reuses the existing `etl/` package (warehouse access, config) and
the existing SQLite database; it only *adds* the `chunks` / `vec_chunks` tables.

## 2. Component design

**Ingestion (offline).** Load PubMedQA → chunk **per `CONTEXTS` paragraph** (carrying
its `LABELS` section) → embed each chunk with `S-PubMedBert-MS-MARCO` → store vector +
text + source metadata (incl. PMID, `final_decision`, `YEAR`). One CLI run builds the
index over all 40 records (~128 chunks).

**Security gate.**
- Rule-based: regex for prompt-injection (`ignore previous…`), PII, and unsafe asks
  (e.g. requests for personalized dosing/diagnosis we shouldn't answer).
- Embedding-based: cosine similarity of the query to a small set of "unsafe / off-topic"
  exemplar embeddings; over a threshold ⇒ block with a safe refusal.

**Domain gate.** Is this a healthcare/facility question at all?
- Rule-based: domain keyword/pattern check (fast accept).
- Embedding-based: max similarity to the corpus; below a floor ⇒ **out-of-domain**.
- **Policy:** out-of-domain (not healthcare) ⇒ **polite refusal**. Web search is *not*
  the out-of-domain path — it is reserved for in-domain queries whose top RAG
  similarity falls below `RAG_THRESHOLD` (see router T3).

**Intent router (tiered, matches the brief's ordering).**
- **T1 rule-based → structured/SQL:** patterns like a 6-digit CCN, or words like
  `rating / rank / beds / completeness / history` ⇒ answer from the **warehouse** via
  **parameterized SQL templates** (fixed schema; text-to-SQL out of scope).
  Deterministic, cheap, no injection surface, unit-testable.
- **T2 embedding → RAG:** embed the query, retrieve top-k chunks. If top cosine
  similarity ≥ `RAG_THRESHOLD`, proceed to the response stage.
- **T3 fallback → web search:** if T2 is below threshold (or domain gate said
  out-of-corpus-but-in-domain), use Claude's `web_search` tool.

**Response (2-stage, line 6 of the brief).**
- **a) Shorten:** extractive compression — rank sentences in the retrieved chunks by
  similarity to the query, keep the top-N. Free, deterministic, trims noise/tokens.
  (Mostly *pedagogical* given Claude's context window — implemented to honor brief
  line 6 and cut tokens, not relied on for answer quality.)
- **b) Answer:** Claude Opus 4.8 receives the shortened context and produces a
  grounded answer **with inline citations** to the source chunks; refuses cleanly when
  context is insufficient.

**Frontend.** A Chat tab in the React app: message list, input, streaming tokens,
and a citations panel. Talks only to the FastAPI `/chat` endpoint (no API key in browser).

## 4. Build sequence (phases)

1. **Phase 1 — Ingestion, vector store & calibration.** `embeddings.py` (load-once
   model) ✓, `store.py` (sqlite-vec schema + cosine search, **PMID in metadata**) ✓,
   `ingest.py` CLI (chunk PubMedQA contexts per paragraph; + dim assertion +
   `final_decision`/`YEAR` columns), `eval.py` (retrieval recall@k/MRR + **analytical**
   covered-vs-uncovered `RAG_THRESHOLD` calibration — no physical hold-out). Verify:
   ingest the sample, run a similarity query, get sensible neighbors, and produce a
   *measured* threshold. Tests for chunking + search ✓ + eval metric math.
2. **Phase 2 — Gates & router (pure Python core).** `security.py`, `domain.py`,
   `router.py`, plus `sql_answer.py` for structured questions. Unit-test the
   rule/embedding decisions with fixtures (no network, no LLM).
3. **Phase 3 — Response pipeline.** `compress.py` (extractive shorten) + `generate.py`
   (Claude Opus 4.8, streaming, citations, `web_search` fallback) + `pipeline.py`
   orchestration. Verify end-to-end from the CLI.
4. **Phase 4 — FastAPI backend.** `app.py` exposes `POST /chat` with SSE streaming;
   wires the pipeline; keeps the Claude key server-side.
5. **Phase 5 — React Chat frontend.** `ChatPanel.jsx` tab: streaming answer +
   citations panel; talks to the backend.

Each phase is independently testable; the pure-Python gates/router/compression are
unit-tested offline, and Claude calls are isolated in `generate.py`.

## 5. Cross-cutting notes

- **Claude usage:** `claude-opus-4-8`, adaptive thinking, **streaming** (long answers);
  `web_search` server-side tool for T3; the official `anthropic` Python SDK. Key from
  an env var on the backend only — never in the browser or committed.
- **Reuse:** shares the existing SQLite DB and `etl.config`; structured answers query
  the warehouse tables we already built.
- **Safety posture:** the system answers *informational* medical questions with
  citations and refuses individualized diagnosis/treatment requests — appropriate for a
  QA/compliance healthcare context.
- **Testing:** keep gates/router/compression pure and unit-tested; isolate I/O
  (embeddings load, DB, Claude) in thin modules.
- **Backups:** take a fresh zip backup before starting the build.
